# 数字工程生态系统（DEE）概述

## 摘要

数字工程（Digital Engineering）是美国国防部推行的工程方法转型，核心是从**基于文档**转向**基于模型**。2023年DoDI 5000.97将MOSA列为数字工程九大要素之一，标志着MOSA从采办策略扩展为数字工程的技术基础。

## 演进脉络

| 时间 | 里程碑 | 意义 |
|------|--------|------|
| 2008 | INCOSE MBSE方法论调查 | 系统梳理主流MBSE方法 |
| 2018 | DoD数字工程战略 | 纲领性文件，定义五大目标 |
| 2023 | DoDI 5000.97 | 指令性文件，定义九大要素 |
| 2024 | DSB数字工程报告 | 评估进展，识别瓶颈 |
| 2024 | Chris Garrett AFMC DMM框架 | AFMC级DE实施框架 + GRA系统（79文档→36模型） |

## DoDI 5000.97 九大要素

| # | 要素 | 说明 | 与MOSA关系 |
|---|------|------|-----------|
| 1 | **权威数据源** | 单一真实来源 | MOSA接口规范的基础 |
| 2 | **数字线程** | 贯穿生命周期的数据流 | MOSA模块间的数据贯通 |
| 3 | **数字孪生** | 物理系统的虚拟镜像 | MOSA模块的虚拟验证 |
| 4 | **MBSE** | 基于模型的系统工程 | MOSA架构的模型化设计 |
| 5 | **MOSA** | 模块化开放系统方法 | **核心要素** |
| 6 | **自动化T&E** | 虚拟+物理联合测试 | MOSA模块的持续验证 |
| 7 | **DevSecOps** | 持续集成/部署 | MOSA模块的开发流程 |
| 8 | **数据治理** | 质量、可访问性、安全 | MOSA接口数据管理 |
| 9 | **DEE基础设施** | 技术环境支撑 | MOSA开发的基础设施 |

## 核心方法论：MBSE

MBSE（基于模型的系统工程）是DEE的方法论引擎。INCOSE 2008年调查定义的主流方法：

| 方法论 | 来源 | 核心思想 |
|--------|------|---------|
| OOSEM | Lockheed Martin / INCOSE | 面向对象SE，基于SysML |
| Harmony-SE | IBM Rational | RUP在SE领域的扩展 |
| Vitech MBSE | Vitech Corporation | 基于CORE工具 |
| JPL State Analysis | NASA JPL | 状态驱动 |

MBSE的核心特征：
1. 模型是主要工件（不是文档）
2. 可执行模型（可仿真、可验证）
3. 单一模型库（所有利益相关者基于同一模型）
4. 自动化追溯（需求→设计→测试）

## 国际标准：ISO 15288:2023

ISO/IEC/IEEE 15288:2023（第二版）定义了系统生命周期过程的4类过程组：
- **协议过程**：获取、供应
- **组织项目使能过程**：生命周期模型管理、基础设施管理、项目组合管理、人力资源管理、质量管理
- **技术过程**：利益相关者需求定义、系统需求定义、架构定义、设计定义、系统分析、实现、集成、验证、确认、转换、使用、维护、处置
- **项目过程**：项目规划、项目评估与控制、决策管理、风险管理、配置管理、信息管理、测量

## DSB 2024年关键发现

国防科学委员会（DSB）2024年报告确认：
1. **MOSA是DEE的技术基础** — 没有开放接口，模型无法互操作
2. **文化变革是瓶颈** — 从"基于文档"到"基于模型"需要组织变革
3. **MBSE工具碎片化** — 缺乏统一建模标准
4. **数字孪生依赖MOSA** — 虚拟验证需要模块化接口定义

## NASA数字工程实施体系（2020-2025）

NASA构建了完整的数字工程实践体系，覆盖采办、建模和质量管理三个维度：

| 维度 | 文档 | 核心内容 |
|------|------|---------|
| **采办框架** | [[NASA手册1004]] | DRD模板、合同语言、MBE计划、互操作性标准（217页） |
| **系统建模** | [[NASA手册1009A]] | SysML元模型、SE引擎对齐、MBSE Grid框架（88页） |
| **M&S质量** | [[NASA手册7009A]] | M&S生命周期管理、V&V体系、置信度评估（157页） |
| **实施进展** | [[IEEE-NASA数字工程之旅2024]] | 692个MBSE用户、1100+模型、8维度能力评估 |

NASA DE的关键实施数据（FY23末）：
- **692个唯一MBSE用户**遍布所有中心和5个任务局
- **1100+个模型**在OCE Teamwork Cloud上创建和管理
- **NPR数据驱动分析**发现860个Blocks、460个Terms、49处差异
- **5级MBSE培训体系**从管理者到专家建模者

与DoD DE体系的对比：NASA更注重实践驱动（志愿者行动→机构部署），DoD更注重政策驱动（立法→指令→实施）。

## AFMC数字化物资管理（DMM）

[[空军装备司令部|AFMC]]（空军装备司令部）正在推动数字化物资管理（[[数字装备管理|DMM]]）框架，这是DEE在AFMC层面的具体实施。DMM与DEE在目标、方法和挑战上高度趋同，但视角不同：DEE强调工程方法论和技术标准，DMM强调组织变革和劳动力赋能。

### DMM六大战略方向

DMM框架由六个战略方向构成：

1. **制定数字战略**：摒弃陈旧实践，为敏捷采办和保障铺路
2. **灌输数字优先文化**：在不断变化的威胁环境中革新AFMC业务方式
3. **提供DMM工具访问**：为数字作战配备动态工具箱
4. **现代化IT基础设施**：持续支持快速企业解决方案
5. **培训数字劳动力**：在全数字生态系统中与合作伙伴协作
6. **结构化和安全化数据**：低摩擦的跨组织团队合作和决策

### GRA系统：从文档到模型

DMM最具操作性的成果是政府参考架构（[[政府参考架构|GRA]]）体系，实现了从文档到模型的转换：

| 维度 | 详情 |
|------|------|
| **文档基线** | 79个MIL标准文档 |
| **模型化** | 转换为36个系统模型 |
| **工具链** | MBSE模型库 + OCE Teamwork Cloud |
| **目标** | 建立可复用、可验证的政府参考架构 |

GRA体系的量化成果（79文档→36模型）是DMM最具说服力的实施证据，也是[[数字装备管理|DMM]]与[[数字工程生态|DEE]]趋同的具体体现。

### 数字生态系统

DMM的工具和平台层由四个核心系统组成：

| 系统 | 功能 |
|------|------|
| **PLM** | 产品生命周期管理 |
| **FENCES** | 网络与安全基础设施 |
| **CLOUDONE** | 云环境与协作平台 |
| **PLATFORMONE** | DevSecOps平台，支持CI/CD |

### 关键挑战

DMM实施面临的挑战与DEE高度重叠，反映了数字化转型的共性障碍：

1. **供应商锁定**：工具互操作性和软件供应商锁定——直接映射[[供应商锁定|MOSA反锁定目标]]
2. **数据标准**：数据标准、格式、建模风格指南的统一
3. **遗留过渡**：遗留项目和流程的数字化过渡——这是最具技术难度的挑战
4. **军种集成**：USAF vs USSF vs DAF的集成和共性
5. **安全扩展**：跨域解决方案/多级安全（CDS/MLS）的可扩展性
6. **合同机制**：数据项描述、合同数据需求清单、IP/数据权利、实时数据访问

### 与DEE的关系

DMM是AFMC层面的DEE实施，两者在目标和方法上高度趋同但视角不同：

| 维度 | DMM（AFMC层面） | DEE（DoD层面） |
|------|-----------------|----------------|
| **发布机构** | AFMC / AFLCMC | OSD(R&E) / DoD |
| **侧重点** | 组织变革、劳动力赋能 | 工程方法论、技术标准 |
| **操作性** | 更具体（六大战略方向） | 更抽象（五大目标） |
| **架构实践** | 有具体GRA建设成果（79→36模型） | 提供方法论框架 |
| **文化维度** | "数字优先文化"为核心支柱 | 文化变革为五大目标之一 |

两者的趋同表明：数字化转型既需要工程方法论（DEE），也需要组织变革框架（DMM）。参见[[DMM与DEE融合]]获取详细对比分析。

## DoD数字工程战略2018 五大目标

1. **权威数据源环境** — 建立单一真实来源
2. **端到端数字工程能力** — 数字线程贯穿生命周期
3. **基于模型的方法** — 增强创新和技术成熟度
4. **数字工程文化变革** — 组织和流程转型
5. **数字工程基础设施** — 技术环境支撑

## 与传统方法的对比

| 维度 | 传统方法 | 数字工程 |
|------|----------|----------|
| 设计产出 | 文档（ICD、SRS） | 模型（SysML、数字孪生） |
| 接口定义 | 纸质ICD | 数字化接口规范 |
| 验证方式 | 事后集成测试 | 虚拟验证+物理联合测试 |
| 供应商交互 | 文档交换 | 模型共享 |
| 升级方式 | 整体替换 | 模块化独立升级 |
| 数据贯通 | 手工追溯 | 数字线程自动追溯 |

## 实践挑战

1. **工具碎片化** — MBSE工具（Cameo、Rhapsody、Capella）互操作性差
2. **人才缺口** — 传统工程师习惯文档驱动，MBSE需要建模技能
3. **投资回报** — 数字工程前期投入大，效益需多年显现
4. **安全与开放的张力** — 参见[[网络安全与模块化开放性]]
