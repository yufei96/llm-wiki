# 新项目MOSA实施操作步骤

## 概述

本文整合DoD 2025指南、NAVAIR OSMP框架、海军MOSA指南和NDIA白皮书，提供从立项到退役的全流程操作步骤。以WOSA（武器开放系统架构）为例说明模块分解和接口所有权划分。

> 核心原则：**MOSA是业务+技术策略，不是纯技术问题。** 政府控制接口和行为的"契约"，供应商自由竞争模块内部的"秘方"。

---

## 第一阶段：架构分解与商业模型（立项/MSA阶段）

### 步骤1：明确MOSA目标与范围

**要回答的问题：**
- MOSA要解决什么？竞争性采购？快速技术刷新？跨平台互操作？
- 选哪种深度——全面模块化（Extensive）还是选择性模块化（Selective）？

NAVAIR框架区分三种设计（见Appendix B图示）：

| 方式 | 适用场景 | 代价 |
|------|---------|------|
| **全面模块化开放** | 全新项目，长期演进需求强 | 前期投入高，治理复杂 |
| **选择性模块化开放** | 现役系统改造，预算有限 | 灵活性低于全面模式 |
| **专有封闭设计** | 仅特定情况可接受 | 供应商锁定，升级困难 |

**输出**：MOSA策略声明，纳入采办策略文档。

### 步骤2：业务案例分析与AoA

- 备选方案分析中完整评估MOSA对每个方案的影响
- **估算MOSA对全寿命周期成本的影响**（重点：维护成本占~70%）
- ⚠️ **GAO-25-106931发现**：20个项目中无一做正式MOSA成本效益分析——这是最大短板

### 步骤3：模块化分解

**总原则（海军指南§4.5-4.7）**：先功能分解，再物理分解。模块内高内聚，模块间松耦合。

**功能分解**的出发点——以WOSA为例：

| 来源 | 分解维度 | 具体要素 |
|------|---------|---------|
| Purdue论文（学术基础） | 5个架构因素 | 毁伤机制、引信、武器重量、制导、推进 |
| Raytheon（工业实践） | 9个标准化子系统 | 机体、航电、执行机构、传感器、软件、推进、电源、有效载荷、通信 |

实际使用时可分层：顶层按Raytheon的9子系统，战斗部层级按Purdue的5因素细化。

**物理分解的注意事项**（海军指南§4.7）：
- 多软件模块跑在同一板卡上 → **所有软件间接口仍然必须暴露**
- 物理上合并可以，但接口暴露要求不变
- 安全关键/核安全功能建议独立封装（隔离），减少回归测试需求

## 第二阶段：接口定义与标准选择（TMRR阶段）

### 步骤4：识别关键接口

区分两类交互（海军指南§4.6）：
- **模块内交互（intra-modular）** → MOSA不管，供应商内部的事
- **模块间交互（inter-modular）** → **MOSA的核心**，这些交互的集合 = 接口

**WOSA下的关键跨模块接口**：

```
制导 ←→ 传感器：目标数据（角度、距离、速度、图像）
制导 ←→ 执行机构：舵偏指令
制导 ←→ 推进：推力/推力矢量指令
制导 ←→ 引信：最佳起爆时刻
制导 ←→ 通信：数据链更新
电源 ←→ 所有模块：电压/功率分配
机体 ←→ 挂架：UAI通用武器接口（最关键！）
```

**UAI（通用武器接口）**是WOSA中最重要的单一接口——武器与平台的机械/电气/数据边界。

每个接口需定义四层（NAVAIR OSMP要求）：
1. 物理层：连接器型号、引脚定义
2. 协议层：通信协议（MORA、DDS、OAS REST等）
3. 数据层：数据格式、编码、更新频率
4. 行为层：状态机、时序约束、故障模式

### 步骤5：选择开放标准

优先选用已有军用/商用开放标准（NDIA评分：商用/DoD标准=3分，完整文档=2分，不公开=0分）。

2024年三军备忘录确认的6大已验证标准：

| 标准 | 领域 | 适用场景 |
|------|------|---------|
| **OMS/UCI** | 通用指挥控制接口 | 跨平台指控 |
| **SOSA** | 传感器系统架构 | 雷达/光电/电子战 |
| **AMS GRA** | 敏捷任务套件 | 政府参考架构 |
| **WOSA** | 武器系统架构 | 导弹/弹药 |
| **FACE** | 航空电子软件 | 软件运行时环境 |
| **VICTORY** | 车辆C4ISR互操作 | 地面车辆 |

### 步骤6：建立接口存储库（MSI Repository）

- 集中管理接口定义、版本历史、使用指南
- 所有供应商可透明访问
- ⚠️ GAO发现DoD接口存储库建设不完整，建议统一建设

## 第三阶段：采办策略与合同（TMRR→EMD）

### 步骤7：制定OSMP（开放系统管理计划）

OSMP是"活的文档"（NAVAIR），政府-承包商协作迭代：

```
政府起草初始版 → 招标书中作为合同交付物 → 中标商迭代更新
                                                    ↓
           Milestone A ──── B ──── C ──── 各阶段评审更新
```

OSMP DID关键段（NAVAIR OSMP §3.3-3.4）：
- §3.3.1 设计披露与数据权利策略
- §3.3.3 模块化选择理由归档
- §3.3.4 MOSA支持计划（迭代评估）
- §3.4 接口规范与管理流程
- §3.6 全生命周期管理流程

### 步骤8：IP与数据权利策略——灰盒原则

海军指南§4.4定义了MOSA的IP核心机制：

**灰盒模型（Gray Box Concept）**：

- **政府知道的**：功能（What）、行为、接口 → 开放、标准化
- **政府不需要知道的**：内部实现（How）→ 供应商专有
- **供应商保护的**：实现方法 → 那是"秘方"，创新的激励

**灯泡类比**（海军指南原文）：E26标准灯座只要求灯泡在120VAC下发光。消费者不关心钨丝怎么挤出来的。灯座接口是开放标准，灯泡内部是专有技术。**这就是最早的MOSA。**

### 步骤9：RFP中嵌入MOSA要求

- 招标书明确MOSA技术与非技术原则
- 系统工程计划（SEP）必须包含MOSA技术方案
- 合同数据需求清单（CDRL）纳入OSMP交付
- 评估标准中明确MOSA合规权重

---

## 第四阶段：验证与持续演进（EMD→P&D→O&S）

### 步骤10：验证与确认（VVT）

- 独立测试模块对开放标准的符合性
- 支持模块增量认证（不必等全系统完成）
- 允许预认证供应商方案，加速部署

### 步骤11：持续竞争与技术刷新

- **生产阶段**：开放接口支持多供应商竞争性采购
- **保障阶段**：模块独立升级，技术插入无需全系统重新设计
- **目标**：从"一次性采办"变为"持续能力演进"

## 接口所有权判断标准

| 维度 | 政府拥有/控制 | 供应商专有 |
|------|-------------|-----------|
| 接口标准本身 | ✅ 政府定义并拥有 | — |
| 接口实现（代码/电路） | 可竞争性采购 | 供应商自选实现方式 |
| 模块功能/行为描述 | ✅ 合同中明确 | — |
| 模块内部架构 | 不干预 | ✅ 供应商自由设计 |
| 专有算法/工艺 | 不要求公开 | ✅ 供应商IP |
| 接口版本控制 | ✅ 政府管理（MSI Repository） | 必须兼容 |
| 合规验证 | ✅ 政府/独立三方 | 供应商自证 |

**三问判断法**：
1. 换供应商来做，是否需要这个信息？→ 需要 = 政府必须拥有
2. 不公布这个信息，会不会导致供应商锁定？→ 会 = 政府必须拥有
3. 公开这个信息，会不会摧毁供应商的创新激励？→ 会 = 留给供应商

## WOSA实例：政府必须控制的接口

| 接口 | 原因 |
|------|------|
| UAI（武器-平台接口） | 挂架标准、电气信号、数据协议——防止平台锁定 |
| 制导-引信接口 | 起爆命令格式、时序 |
| 制导-推进接口 | 推力指令、点火信号 |
| 通信链路协议 | Link 16 / MADL等波形规范 |
| 电源分配规范 | 各模块的电压/功率预算 |
| 测试/验证接口 | BIT指令集、自检数据格式 |

## 最易犯的错误

1. **不分析成本效益就上**（GAO头号发现，20项目无一做到）
2. **把MOSA当"技术问题"而非"业务+技术策略"**（NDIA核心观点）
3. **事后拼贴而非从一开始设计**（bolt-on vs built-in）
4. **政府不控制接口**（供应商锁定的根源）
5. **OSMP写完就束之高阁**（应该是活文档，随里程碑迭代）
6. **把物理合并等同于模块合并**（一块板卡多软件→接口仍须暴露）

| 军种 | 实施率 | 指南成熟度 | 典型项目 |
|------|--------|-----------|----------|
| 陆军 | 100% | ✅ 最完善 | FLRAA、PrSM、LTAMDS、IBCS |
