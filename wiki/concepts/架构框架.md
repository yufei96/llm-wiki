# 架构框架

## 定义

架构框架是描述复杂系统结构、行为和约束的形式化方法论与工具集。在国防采办语境中，架构框架为系统设计提供标准化的视角、视点和描述方式，支撑模块化开放系统方法（MOSA）的实施。

## 架构描述的核心要素

1. **结构视图**：系统由哪些组件构成，组件之间如何分层和组织
2. **行为视图**：系统如何运行，组件之间如何交互和协作
3. **约束视图**：系统必须满足哪些需求、标准和接口规范
4. **分配视图**：功能如何映射到物理组件，需求如何追溯到实现

## 国防架构框架演进

| 框架 | 主管机构 | 重点 | 状态 |
|------|---------|------|------|
| **C4ISR / DoDAF** | 国防部 | 指挥控制视角的架构描述 | 成熟使用中 |
| **UPDM / UAF** | OMG | 统一架构框架，跨域通用 | 逐步推广 |
| **SysML v2** | OMG / INCOSE | 系统工程专用建模语言 | 2025年正式发布 |
| **NATO NAF** | 北约 | 联盟互操作性架构 | 北约体系内使用 |

## SysML v2 与架构描述语言

SysML v2（Systems Modeling Language v2）是国际系统工程协会（INCOSE）与对象管理组织（OMG）联合开发的新一代系统建模语言，为架构描述提供了**形式化的架构描述语言**。

### 8种图类型

SysML v2 提供 **8种图类型**，比 SysML v1 的6种多出2种，覆盖更完整的架构描述需求：

| 图类型 | 用途 | v1对应 |
|--------|------|--------|
| **Package图** | 组织模型元素的层级结构 | 类似v1但增强 |
| **Part图** | 描述系统组件的物理/功能分解 | 原Block Definition图 |
| **State图** | 描述系统/组件的状态与转换 | 延续v1 |
| **Action图** | 描述行为、活动和数据流 | 原Activity图，重设计 |
| **Requirement图** | 需求的结构化描述与追溯 | 增强 |
| **Use Case图** | 系统与外部交互的用例描述 | 新增（独立图类型） |
| **Sequence图** | 时序交互与消息交换 | 新增（独立图类型） |
| **Connection图** | 接口与连接关系 | 原Parametric图替代 |

### 文本化表示法

SysML v2 引入了**文本化表示法（textual notation）**，与图形表示法并行。这是架构工程的范式转变：

- **版本控制友好**：架构模型可以像代码一样纳入Git等版本控制系统
- **差异对比（diff）**：可精确识别架构变更的具体内容
- **合并（merge）**：支持多团队并行架构开发，自动/半自动合并
- **可脚本化**：架构描述可通过脚本生成、转换和验证

### API优先的工具互操作

SysML v2 采用 **API优先（API-first）** 设计理念，通过标准化REST API实现工具间互操作：

- **打破工具孤岛**：不同厂商的建模工具可通过统一API交换模型数据
- **vs v1的XML困局**：SysML v1依赖XMI/XML交换格式，实际互操作性极差，各工具形成数据孤岛
- **集成生态**：支持CI/CD管道、自动化测试、需求管理工具等的深度集成

### 架构语言的公共底座

> **SysML v2 之于架构描述，正如 WOSA 之于数据采集——它是架构领域的公共语言。**

WOSA为数据采集提供了统一的工具链和评估标准；SysML v2则为架构描述提供了统一的形式化语言，使得不同项目、不同承包商、不同军种之间的架构模型可以**互通、互审、互操作**。

## 架构框架与MOSA的关系

架构框架是MOSA的**技术实施载体**：

- **接口定义**：架构框架明确定义模块边界和接口规范
- **模块划分**：通过架构分析确定最优模块粒度
- **合规验证**：架构模型支撑MOSA合规性评估（WOSA工具集）
- **竞争基础**：清晰的架构描述是供应商公平竞争的前提
