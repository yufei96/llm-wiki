# 模拟器通用架构需求标准（SCARS）

## 摘要

SCARS（Simulators Common Architecture Requirements and Standards，模拟器通用架构需求标准）是美国空军的一项维持性倡议，旨在为空军模拟器逐步建立模块化开放系统方法（MOSA）。SCARS由HAF/A3T（空军总部训练局）发起，CAE USA担任主承包商，采用IDIQ合同模式（5年基础期2020-2025 + 5个1年选项期至2030）。该项目覆盖50+平台、2380+训练设备、7个主要司令部（MAJCOM），是空军训练基础设施现代化的核心项目。SCARS通过建立通用架构、标准化接口和网络安全框架，实现训练系统的能力复用、网络弹性和全生命周期成本最小化。

## 核心目标

### 问题陈述

空军训练基础设施面临的核心挑战：
- **配置碎片化**：2400+训练设备（50+平台）配置各异，复用性极低
- **开发成本高昂**：每个设备的独特性导致昂贵的独特开发、生产和维护
- **网络安全威胁**：持续演变的网络威胁，RMF（风险管理框架）实施成本高昂
- **训练效能受限**：缺乏标准和通用架构限制了训练效果
- **互操作性不足**：高保真度和互操作性需求日益增长

### SCARS核心目标

1. **建立MOSA**：为空军模拟器增量建立模块化开放系统架构
2. **提升网络弹性**：应对持续演变的网络安全威胁
3. **最小化全生命周期成本**：通过通用架构减少独特开发和维护成本
4. **增强训练效能**：通过标准化和复用提升训练效果
5. **支持分布式训练**：通过网络连接模拟器实现分布式任务训练

## 核心特点

### 三阶段实施方法

SCARS采用渐进式三阶段方法：

**第1阶段：高效网络安全（Efficient Cybersecurity）**
- 提供集中化网络维护：漏洞扫描、补丁、防病毒和企业级RMF控制
- 通用网络安全工具
- 集中化训练设备扫描/补丁管理
- **战士收益**：显著减少训练系统维护停机时间，加速ATO（授权运营）获取

**第2阶段：通用架构（Common Architecture）**
- 训练系统大部分功能迁移到通用基础设施
- 新技术支持向云/虚拟计算的完全迁移
- 通用建模和接口
- 虚拟化非硬件依赖的软件
- **战士收益**：可靠性提升、维护最小化、功能改进成本降低

**第3阶段：标准应用（Standard Applications）**
- 训练系统采用通用应用
- 通用应用托管基础设施
- 仅不可虚拟化的软件为训练器特定
- 广泛的通用模型库
- **战士收益**：跨平台应用复用，一致性性能增强"公平交战"（Fair Fight）

### SCARS关键组件

| 组件 | 功能 |
|------|------|
| **安全运营中心（SOC）** | OPE和SCARS WAN的远程管理、中央仪表板、帮助台 |
| **本地设备（OPE）** | 本地云、可扩展、通用安全控制 |
| **广域网（WAN）** | 利用现有伙伴网络，概念验证阶段 |
| **授权训练系统** | 无ATO不接入SCARS |
| **SCARS标准** | 建立通用架构的关键 |

### SCARS标准体系

SCARS定义了一套完整的标准体系：

| 标准 | 说明 |
|------|------|
| **基础OPE集成标准** | 训练系统与SCARS OPE集成的ICD |
| **通用安全控制标准** | 训练系统利用SCARS网络安全控制的ICD |
| **认证流程标准** | 测试训练系统SCARS合规性的流程 |
| **应用认证标准** | 在SCARS OPE上认证和部署训练软件的流程 |
| **数据标准** | 标准数据格式ICD，含MBSE建模指南和地理空间数据规范 |
| **DevSecOps标准** | 利用SCARS DevSecOps管道进行训练软件维护的流程 |
| **架构标准** | MOSA合规模块化子系统和接口的ICD |

### 采用的行业标准

SCARS积极采纳现有行业标准：
- **OGC CDB**（通用数据库）：地理空间数据交换格式
- **SISO CIGI**（通用图像生成器接口）：模拟器与图像生成器接口
- **IoT**（物联网）：模拟器与机组控制系统接口
- **ARINC-610C**：使用飞机部件时的通用模拟器功能
- **OMG SysML**：模型化系统工程标准
- **Open XR**：扩展现实接口标准

### 架构转变

SCARS推动的架构转变——从碎片化到标准化：

**实施前**（碎片化）：
- 内部和自定义签名/代码
- 自定义接口连接各组件
- 专有运行时
- 过时的源数据
- 每个平台完全独特

**实施后**（标准化）：
- 通用签名和天气数据
- JSE GRID服务处理物理交互
- 通用弹药模型（JSE WSCE）
- 标准气动模型接口
- IoT/GDR/SISO CIGI/OGC CDB标准接口
- Open XR扩展现实标准

## 与其他标准的关系

### 在开放架构层次体系中的位置

SCARS位于[[开放架构层级|开放架构四层体系]]的**第3层：设备级**，是训练模拟领域的MOSA实施。

### 与MOSA的关系

SCARS是[[MOSA与国防采办|MOSA原则]]在训练领域的系统性实施：
- **模块化设计**：训练系统组件模块化
- **指定接口**：标准化内部接口和数据格式
- **开放标准**：采纳OGC、SISO、IoT等行业标准
- **一致性认证**：SCARS认证流程标准
- **赋能环境**：政府拥有标准，促进供应商竞争

### 与其他架构的互补关系

- **SCARS ↔ OMS**：SCARS架构标准中引用OMS作为空气器接口标准
- **SCARS ↔ FACE**：训练系统可复用FACE标准的软件组件
- **SCARS ↔ SOSA**：传感器模拟可采用SOSA硬件标准
- **SCARS ↔ ABMS GRA**：训练系统需模拟ABMS网络环境

### 独特定位

SCARS在MOSA生态中有独特定位：
- **覆盖范围最广**：50+平台、2380+设备、7个MAJCOM，是覆盖范围最广的MOSA实施之一
- **维持性路径**：采用维持性采办路径（非新研），通过增量方式建立MOSA
- **网络安全驱动**：网络安全合规（RMF）是SCARS的首要驱动力
- **企业级方法**：不是单个平台的MOSA，而是整个空军训练基础设施的MOSA

## 笔记

- SCARS的"三阶段渐进方法"是MOSA实施的最佳实践——从网络安全（最紧迫）→通用架构→标准应用，逐步深化
- "无ATO不接入SCARS"政策体现了网络安全在SCARS中的核心地位——安全合规是MOSA的先决条件
- SCARS采用IDIQ合同模式（5年+5年选项），说明MOSA在训练领域的实施是长期持续的过程
- 2400+模拟器、300+站点的规模使SCARS成为国防部最大规模的MOSA部署项目之一
- SCARS标准由AFLCMC/WNS发布（aflcmc.wns.scarsstandards@us.af.mil），部分标准限分发
- OPE（本地设备）+ SOC（安全运营中心）的架构模式类似于企业IT的云+安全运营中心模式
- SCARS的"虚拟化非硬件依赖软件"策略直接解决了传统训练系统硬件绑定的问题
- 行业标准采纳（OGC CDB、SISO CIGI、IoT等）体现了MOSA"利用现有标准"的原则
- SCARS的行业期望：模块化架构、标准接口、标准数据格式、避免专有接口、IoT控制、软件容器
- CAE USA通过收购L3 Link Simulation业务成为SCARS主承包商，体现了训练领域的行业整合趋势
